# Phase 3: 虚拟文件系统与元数据索引

## 1. 阶段目标

在可扩展虚拟磁盘之上建立真正可用的文件系统，包括：

- 目录
- 文件
- 路径解析
- 元数据
- 数据块映射
- 持久化恢复

## 2. 设计选择

本阶段的重点是把文件系统接口、元数据、一致性与持久化做好，而不是手写通用数据结构。

因此目录索引层采用以下原则：

- Tiny OS 必须自己定义目录索引抽象接口
- 目录索引底层可以直接使用成熟 B+ 树库或其他成熟有序索引库
- VFS、路径解析、权限和持久化逻辑不能直接耦合到具体库
- 索引实现必须可替换

推荐做法：

- 定义 `DirectoryIndex` 或等价接口
- 先写基于接口的测试
- 再接入一个成熟索引实现
- 用适配器模式把第三方库封装起来

## 3. 核心对象

建议至少定义：

- `Inode`
- `DirectoryEntry`
- `DirectoryIndex`
- `SuperBlock`
- `FreeBlockMap`
- `VirtualFileSystem`

## 4. 功能需求

### 4.1 路径系统

需要支持：

- 绝对路径
- 相对路径
- `.` 与 `..`
- 根目录 `/`

### 4.2 基础文件系统操作

需要支持内部 API：

- `mkdir(path)`
- `create_file(path)`
- `read_file(path)`
- `write_file(path, data)`
- `delete_file(path)`
- `list_dir(path)`
- `stat(path)`

### 4.3 元数据

至少包含：

- inode 编号
- 类型：文件或目录
- 大小
- 创建时间
- 修改时间
- 权限位
- 所属用户

### 4.4 目录索引接口

目录索引层至少要提供以下抽象能力：

- `get(name)`
- `set(name, inode_id)`
- `delete(name)`
- `list_entries()`
- `contains(name)`

要求：

- `list_entries()` 输出顺序稳定
- 不把第三方库对象直接泄露给 VFS
- 支持后续替换为其他索引后端

### 4.5 大文件支持

文件内容不能只塞在一个对象里，必须映射到多个数据块。

要求：

- 支持跨多个块写入
- 支持覆盖写
- 支持追加写
- 支持恢复后重新读取

## 5. 持久化要求

至少要能把以下状态恢复回来：

- 根目录
- 目录结构
- inode 元数据
- 文件块映射
- 空闲块状态

## 6. TDD 要求

至少覆盖：

- 目录创建测试
- 多级路径解析测试
- 文件创建与读取测试
- 大文件跨块写入测试
- 删除文件后块回收测试
- 重启恢复测试
- 非法路径测试
- 重名冲突测试
- 目录索引适配器测试
- 目录索引输出顺序稳定性测试

如果底层选用 B+ 树或其他复杂有序索引，还必须补充：

- 库适配层回归测试
- 有序遍历测试
- 持久化恢复后的结构一致性测试
- 至少一个覆盖高目录项数量的压力测试

## 7. 验收标准

- Shell 尚未完成前，也能通过内部 API 完整操作文件系统
- 系统重启后仍能恢复目录和文件
- 在有几十到几百个目录项时，查询和遍历仍保持清晰稳定

## 8. 风险

- 路径解析如果没有系统测试，后续 Shell 会出现大量隐蔽 bug
- 元数据和数据块映射如果耦合太乱，后续权限和观测功能很难接入
- 如果 VFS 直接依赖某个索引库的私有行为，后续替换实现会非常痛苦
