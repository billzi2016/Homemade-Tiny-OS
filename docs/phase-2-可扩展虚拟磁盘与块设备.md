# Phase 2: 可扩展虚拟磁盘与块设备

## 1. 阶段目标

建立一个**小起步、按需扩容、可持久化、有硬上限、不会无节制写宿主 SSD** 的虚拟磁盘子系统。

这是整个系统的存储地基。地基如果设计差，后面的文件系统和恢复机制都会崩。

## 2. 核心原则

- 初始镜像小
- 扩容按块进行
- 扩容有上限
- 只在必要时扩容
- 尽量减少宿主机写放大
- 扩容失败时返回明确错误

## 3. 功能需求

### 3.1 磁盘镜像文件

实现一个磁盘镜像文件，例如 `tinyos.disk`。

要求：

- 支持首次创建
- 支持重复打开
- 支持关闭后重新恢复
- 支持元信息校验

### 3.2 容量策略

建议默认值：

- 初始容量：`4 MiB`
- 扩容步长：`1 MiB`
- 块大小：`4 KiB`
- 最大容量：`64 MiB`

这些值必须允许配置覆盖。

### 3.3 扩容机制

当写入所需块数超过当前容量时：

1. 先判断扩容后是否超过最大上限
2. 未超过则按步长扩容
3. 超过则抛出 `DiskFullError`

要求：

- 不一次性预分配巨大空间
- 优先采用稀疏文件或等效策略
- 不因单次小写入触发大量无意义扩容

### 3.4 块设备接口

最少需要以下接口：

- `read_block(block_id) -> bytes`
- `write_block(block_id, data) -> None`
- `allocate_blocks(count) -> list[int]`
- `free_blocks(block_ids) -> None`
- `flush() -> None`

## 4. 数据一致性要求

至少要保证：

- 块大小固定
- 块号非法时拒绝访问
- 部分写入失败后不会让元数据处于未知状态
- 重新打开镜像后能读出已写入数据

## 5. TDD 要求

本阶段必须优先写测试，尤其是边界测试。

至少覆盖：

- 新磁盘初始化测试
- 已有磁盘重新打开测试
- 块读写测试
- 刚好触发一次扩容测试
- 连续多次扩容测试
- 超过上限拒绝测试
- 关闭后恢复测试
- 非法块号测试

建议增加：

- 属性测试或参数化测试，覆盖多种块大小和容量组合

## 6. 验收标准

- 可以创建并打开一个虚拟磁盘镜像
- 可以稳定读写块
- 容量可以按需增长
- 超出上限会报错，不会继续写
- 测试可以证明磁盘扩容和恢复正确

## 7. 失败风险

- 如果用无限扩容，后续很容易误写大量宿主机磁盘
- 如果块分配和元数据不清晰，Phase 3 文件系统无法稳定建立
