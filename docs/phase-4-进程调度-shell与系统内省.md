# Phase 4: 进程调度、Shell 与系统内省

## 1. 阶段目标

让 Tiny OS 从“有内部能力”变成“人可以真实操作和观察”的系统。

这一阶段要完成三件事：

- 协作式进程调度
- Shell 命令行交互
- 系统内省与监控能力

## 2. 进程调度需求

### 2.1 进程模型

建议以生成器或协程模拟用户态任务。

PCB 至少包含：

- `pid`
- `ppid`
- `name`
- `status`
- `cwd`
- `uid`
- `time_slice`
- `cpu_ticks`

### 2.2 状态机

至少支持：

- `READY`
- `RUNNING`
- `BLOCKED`
- `ZOMBIE`

### 2.3 调度器

首版建议 Round-Robin。

要求：

- 每轮只推进一个最小执行片段
- 任务主动 `yield`
- 进程结束后进入 `ZOMBIE`
- Shell 可以查看当前队列状态

## 3. Shell 需求

### 3.1 命令约束

所有命令必须调用 Tiny OS 自己的内部 API，不允许透传宿主机命令。

### 3.2 首批命令

至少支持：

- `ls`
- `cd`
- `pwd`
- `mkdir`
- `touch`
- `cat`
- `echo`
- `rm`
- `ps`
- `kill`

### 3.3 命令解析

至少支持：

- 命令 + 参数
- 引号包裹的文本
- 相对路径与绝对路径
- 基础重定向，例如 `echo "text" > file`

## 4. 系统内省需求

本项目必须加入“能看见系统在活着”的观测能力。

### 4.1 `top` 或 `mon`

要求展示：

- 总 tick 数
- 当前运行进程
- 就绪队列
- 各进程状态
- 当前工作目录

### 4.2 `sysstat`

要求展示：

- 虚拟磁盘容量、已用容量、剩余容量
- 块分配情况
- 内存页使用情况
- 文件系统索引概览
- 关键计数器，例如页错误、树分裂、调度切换次数

### 4.3 `dmesg`

要求维护一个固定容量的内核日志环形缓冲区，记录例如：

- 进程切换
- 磁盘扩容
- 文件写入
- 树分裂
- 权限拒绝

## 5. TDD 要求

至少覆盖：

- 调度器轮转测试
- PCB 状态流转测试
- 进程退出测试
- Shell 命令解析测试
- 命令调用 VFS API 的集成测试
- `top`/`sysstat` 输出快照测试
- `dmesg` 容量上限与覆盖测试

## 6. 验收标准

- 用户可以在交互界面里创建目录、写文件、查看进程
- 可以看到进程轮转和系统指标
- 命令错误能转化为稳定、可理解的 Shell 输出

## 7. 风险

- 如果 Shell 直接绕过内核访问底层对象，后续权限和审计会失效
- 如果观测接口只做“假打印”，这个项目会失去最有价值的学习部分
