# Git Workflow

## 1. 目标

本文件定义 Tiny OS 项目的 Git 使用方式。

核心原则不是“写一点就 commit 一次”，而是以**可验证的稳定节点**作为提交边界。  
对本项目来说，最重要的稳定节点有三类：

- Phase 阶段完成
- 关键测试点转绿
- 核心接口或架构边界定型

## 2. 初始化流程

在项目刚开始时，建议使用以下节奏：

1. `git init`
2. 提交第一版基础文档
3. 建立 `.gitignore`
4. 建立项目骨架后再提交一次

首个提交建议只包含：

- `README`
- `docs/`
- `.gitignore`
- 其他不会引入实现噪音的初始化文件

## 3. 提交粒度

### 3.1 必须提交的节点

以下节点应当明确提交：

- 一个 Phase 完成且测试通过
- 一个关键测试点从失败变为通过
- 一个重要抽象接口稳定下来
- 一次大重构开始前
- 一次大重构完成且测试回绿后

### 3.2 推荐提交的节点

以下节点通常也值得提交：

- 第一批测试骨架建立完成
- 一个子模块最小可运行版本完成
- 一个新的错误路径被完整测试并修复
- 一个目录结构或持久化格式被正式定型

### 3.3 不建议提交的状态

以下状态不建议提交，除非你有意保留实验现场：

- 改动很杂，彼此无关
- 测试已经明显失败，但还没定位问题
- 一个 commit 同时混入实现、重构、清理和无关文档
- 只是临时打印、临时调试代码

## 4. 推荐节奏

### 4.1 文档阶段

适合先有一笔：

- `初始化项目文档与阶段规划`

### 4.2 Phase 推进阶段

每个 Phase 推荐至少两到四笔提交：

1. 先补测试
2. 最小实现让核心测试转绿
3. 必要重构
4. Phase 收尾与文档同步

### 4.3 关键测试点阶段

对本项目，以下测试点值得单独形成提交边界：

- 虚拟磁盘初始化测试转绿
- 扩容与上限保护测试转绿
- 文件系统基础读写测试转绿
- 重启恢复测试转绿
- 调度器轮转测试转绿
- Shell 基础命令测试转绿
- 权限与空间不足异常测试转绿

## 5. Commit 信息规范

commit 信息必须使用中文，且要直接说明本次变更的核心结果。

推荐格式：

- `文档: 补充 Git 工作流说明`
- `测试: 增加虚拟磁盘初始化与扩容用例`
- `存储: 实现可扩展虚拟磁盘与块分配`
- `文件系统: 完成目录创建与基础读写`
- `调度: 实现 Round-Robin 轮转逻辑`
- `Shell: 接入基础命令解析与分发`
- `可靠性: 增加磁盘写满与权限拒绝处理`

不推荐：

- `update`
- `fix`
- `wip`
- `改一下`
- `试试`

## 6. 与 TDD 的关系

本项目采用 TDD，因此提交顺序应尽量体现出测试驱动过程。

推荐顺序：

1. 提交测试骨架
2. 提交最小实现
3. 提交重构

如果不想把“红测试”提交进主线，至少也要做到：

- 测试和实现在同一小闭环内完成
- commit 边界依然围绕“某一组测试转绿”

## 7. 与 Phase 的关系

建议把 `tasks.md` 中每个 Phase 的完成条件，当作 Git 里程碑判断标准。

示例：

- `Phase 1` 提交时，应至少保证项目骨架、配置、错误模型和测试基线已就位
- `Phase 2` 提交时，应至少保证磁盘初始化、块读写、扩容、上限保护测试通过
- `Phase 3` 提交时，应至少保证 VFS 基础操作和恢复测试通过
- `Phase 4` 提交时，应至少保证调度器、Shell 基础命令和内省命令可用
- `Phase 5` 提交时，应至少保证权限、异常和恢复路径具备基本稳定性

## 8. Tag 建议

在达到明显里程碑后，可以打标签：

- `v0.1-docs`
- `v0.2-phase1`
- `v0.3-phase2-storage`
- `v0.4-phase3-vfs`
- `v0.5-phase4-shell`
- `v0.6-phase5-reliability`

## 9. 简化执行规则

如果你不想每次都想“该不该 commit”，就按下面规则执行：

- 文档成体系了，提交
- 一组关键测试转绿了，提交
- 一个 Phase 完整结束了，提交
- 大重构前后，各提交一次

这套规则足够简单，也足够适合 Tiny OS 这种阶段性很强的项目。
