# Tiny OS PRD

## 1. 文档目标

本文档定义 `Homemade-Tiny-OS` 的产品目标、边界、架构方向、测试纪律与阶段性实施路线。

本项目的目标不是做一个只能演示截图的玩具，而是做一个**运行在宿主机用户态、但内部机制完整、自洽、可测试、可扩展、可持久化的微型操作系统仿真器**。

它不是裸机操作系统，不直接启动硬件；但它必须在以下方面尽量接近真实 OS：

- 自己实现命令解释，不借宿主机系统命令作弊
- 自己维护磁盘、文件系统、进程、调度、内存、权限等核心状态
- 能冷启动、运行、保存状态、恢复状态
- 有严格测试，避免随着功能增加而垮塌

## 2. 产品定位

### 2.1 项目定位

Tiny OS 是一个用于深入理解操作系统原理的工程化项目，面向以下目标：

- 练手操作系统核心抽象，而不是只背概念
- 建立一个适合 AI Agent 分阶段协作开发的硬核项目
- 保证后续能够持续扩展，而不是第一版写完后不可维护

### 2.2 非目标

以下内容不在当前 PRD 的首期目标内：

- 裸机启动
- 中断控制器、真实驱动、真实 MMU
- 多核并行
- 真正的 ELF 装载器
- 对宿主机文件系统做直接映射

## 3. 设计原则

### 3.1 真实感优先

所有用户可见命令必须落到 Tiny OS 自己的内部 API 上，不允许通过 `os.system()`、`subprocess` 等方式调用宿主机命令来伪装成功能。

### 3.2 可测试性优先

本项目必须采用 TDD 思路推进。

- 先写失败测试，再补实现
- 每个 Phase 都必须定义单元测试、集成测试、验收测试
- 当前 Phase 的测试未通过，不进入下一 Phase
- 对关键数据结构引入边界测试和回归测试

### 3.3 持久化与安全边界优先

虚拟磁盘必须支持按需扩展，但不能无上限增长，更不能粗暴持续写宿主 SSD。

默认策略要求：

- 初始磁盘镜像较小，例如 `4 MiB`
- 按固定块增量扩展，例如每次 `1 MiB`
- 有默认硬上限，例如 `64 MiB` 或 `128 MiB`
- 支持配置上限
- 优先使用稀疏文件或等价的低写放大策略
- 只有在实际需要时才扩容
- 超限时返回明确错误，不得继续写入

### 3.4 模块边界清晰

各模块必须通过明确接口协作，禁止 Shell 直接操作底层存储结构，禁止调度器越层修改文件系统内部结构。

### 3.5 实现粒度约束

本项目是教学型工程项目，因此要避免两个极端：

- 什么都自己造，导致项目被通用底层细节拖垮
- 什么都直接 import，导致失去操作系统学习价值

因此实现粒度必须明确如下。

#### 必须自己实现

以下能力属于 Tiny OS 的核心教学目标，必须自己设计并实现：

- 虚拟磁盘抽象
- 磁盘扩容策略与容量上限控制
- 块分配与块回收策略
- VFS 对外接口
- 路径解析与命名空间规则
- `Inode`、目录项、超级块等核心元数据模型
- 文件内容到数据块的映射逻辑
- 进程模型、PCB 和调度器状态机
- Shell 命令解析与命令分发
- 权限模型与错误模型
- 系统观测、指标统计和内核日志

#### 允许使用成熟库

以下能力可以直接使用成熟库，但必须封装在自己的模块后面：

- B+ 树或其他目录索引结构
- 有序映射、缓存、堆等通用数据结构
- CLI 美化与终端输出辅助
- 日志格式化
- 序列化辅助
- 测试辅助工具

要求：

- 上层代码只能依赖项目内部抽象接口
- 选型理由必须能说明
- 更换底层库时，不应大面积破坏上层逻辑

#### 禁止直接外包给库

以下能力不允许整体交给第三方库完成：

- 整个文件系统
- 整个调度器
- 整个 Shell
- 整个权限系统
- 整个 Tiny OS “内核”

#### 索引模块原则

以目录索引为例：

- 项目可以直接采用成熟 B+ 树库
- 但 Tiny OS 必须自己定义 `DirectoryIndex` 之类的抽象接口
- VFS 只能依赖抽象接口，不直接依赖具体库实现
- 后续如果替换为别的索引实现，VFS 与 Shell 不应重写

## 4. 用户故事

### 4.1 作为学习者

我希望能在一个可运行的系统里看到进程切换、目录创建、文件写入、磁盘扩容和权限拒绝，而不是只看静态代码。

### 4.2 作为开发者

我希望每个核心模块都有可单独测试的边界和接口，这样我增加新功能时不会把旧功能打坏。

### 4.3 作为 AI Agent 协作对象

我希望项目可以按 Phase 拆分，每一阶段的目标、约束、接口、测试与验收都明确，便于稳定推进。

## 5. 总体架构

```text
+------------------------------------------------------------+
| Shell / CLI / Monitor                                      |
| ls, cd, mkdir, cat, echo, ps, kill, top, sysstat, dmesg   |
+------------------------------------------------------------+
| Kernel Core                                                |
| Scheduler | Process Manager | Memory Manager | Metrics     |
+------------------------------------------------------------+
| Virtual File System                                        |
| Namespace | Inode | Directory Index | File Block Mapping   |
+------------------------------------------------------------+
| Virtual Storage Layer                                      |
| Expandable Disk Image | Block Allocator | Persistence      |
+------------------------------------------------------------+
| Host OS + Python Runtime                                   |
+------------------------------------------------------------+
```

## 6. 测试策略总则

建议从一开始建立以下测试分层：

- `tests/unit`：核心数据结构、算法、解析器、状态机
- `tests/integration`：磁盘 + 文件系统、调度器 + Shell、持久化恢复
- `tests/e2e`：以命令输入和系统输出为中心的场景测试

关键测试要求：

- 目录索引模块必须覆盖插入、查找、遍历、删除、恢复
- 如果所选索引库暴露分裂、合并或持久化边界，也要写回归测试保护这些行为
- 磁盘扩容必须覆盖未扩容、刚好扩容、超限拒绝、重启恢复
- Shell 命令必须覆盖正常路径、非法参数、权限失败、空间不足
- 调度器必须覆盖 READY/RUNNING/BLOCKED/ZOMBIE 状态切换
- 所有错误码和错误消息都要有回归测试

## 8. 成功标准

项目达到以下条件，视为首个可用版本成立：

- 能创建、读取、覆盖、删除虚拟文件
- 能跨进程或跨会话保留磁盘状态
- 能通过 Shell 完整操作主要功能
- 能看到基础系统观测信息，例如调度、页错误、索引行为、磁盘使用
- 在测试集通过的前提下迭代新功能
- 在空间不足、权限不足、路径非法等异常下保持稳定

## 9. 风险与约束

### 9.1 最大风险

- 功能堆叠过快，缺少测试，导致系统状态机失控
- 直接把宿主机能力当作系统能力，破坏项目纯度
- 虚拟磁盘设计不当，导致扩容不可控或写放大过高
- 把索引实现细节当作首要目标，反而拖慢真正的 OS 核心功能
- Shell 逻辑绕过内核 API，造成接口边界崩坏

### 9.2 约束要求

- 每一阶段结束前必须补齐该阶段测试
- 不允许为了“看起来能跑”而跳过持久化一致性
- 不允许把“模拟输出”冒充“真实内部状态”

## 10. 交付建议

后续开发建议始终遵守以下节奏：

1. 先锁定一个 Phase 的输入输出边界
2. 先写测试
3. 再写最小实现
4. 跑测试修正
5. 最后补文档和演示命令

这样能最大程度避免项目后期结构性崩塌。
