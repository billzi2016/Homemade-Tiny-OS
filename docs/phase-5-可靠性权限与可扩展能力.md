# Phase 5: 可靠性、权限与可扩展能力

## 1. 阶段目标

在系统已经可运行的前提下，补齐它作为“可持续演进项目”最关键的防崩层：

- 错误处理
- 权限隔离
- 空间保护
- 恢复能力
- 后续扩展接口

## 2. 可靠性需求

### 2.1 空间不足保护

当磁盘达到上限、无法继续扩容时：

- 必须拒绝写入
- 必须返回明确错误
- Shell 应显示类似 `No space left on device`
- 系统状态不能被破坏

### 2.2 崩溃恢复

至少要保证：

- 正常关闭后能恢复
- 异常中断后不至于整个镜像不可读
- 元数据损坏时能给出明确错误而不是静默读错

### 2.3 一致性

要求逐步建立最小一致性机制，例如：

- 写前校验
- 元数据版本号
- 超级块签名
- 必要时的简化日志或检查点

## 3. 权限模型

建议最少支持：

- `root` 用户
- 普通用户

文件系统节点至少要有：

- `uid`
- `gid` 或简化后的组概念
- 基础读写执行权限位

需要支持：

- 普通用户无权写受保护目录
- root 可绕过大部分限制
- Shell 能显示权限拒绝

## 4. 可扩展性需求

为后续能力保留接口，例如：

- 插件式命令注册
- 新的调度算法替换
- 新的目录索引结构替换
- 新的持久化格式升级

要求：

- 索引库替换只应影响适配层
- Shell 命令扩展不应破坏已有命令
- 底层持久化格式升级应有版本识别策略

## 5. TDD 要求

至少覆盖：

- 磁盘写满测试
- 权限拒绝测试
- root 放行测试
- 损坏元数据识别测试
- 日志或检查点恢复测试
- 新命令注册兼容性测试
- 索引实现替换后的兼容性测试

## 6. 验收标准

- 在异常路径下系统仍然保持可解释、可恢复、可测试
- 权限行为稳定，不会随机绕过
- 后续加入新命令或新模块时，不需要推翻现有架构

## 7. 版本出口定义

本阶段完成后，可以将 Tiny OS 视为一个首个“可持续演进版本”，具备：

- 可运行
- 可持久化
- 可测试
- 可观测
- 可受控扩展

这时再继续加 IPC、管道、更多命令、更多调度算法，才是健康扩展，而不是继续堆不稳定功能。
